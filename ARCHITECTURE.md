## 1. 架构设计理念

### 1.1 设计目标

| 目标 | 指标 | 说明 |
|------|------|------|
| 高可用 | 99.9% | 年度停机时间 < 8.76 小时 |
| 高性能 | P99 < 200ms | 列表接口响应时间 |
| 可扩展 | 水平扩展 | 支持多实例部署 |
| 可维护 | 模块解耦 | 独立部署、独立演进 |
| 安全合规 | 等保二级 | 数据脱敏、审计日志 |

### 1.2 架构原则

```
┌─────────────────────────────────────────────────────────────────────┐
│                      架构设计六大原则                                │
├─────────────────────────────────────────────────────────────────────┤
│  1. 简单够用     不过度设计，用最简方案解决实际问题                   │
│  2. 演进式      预判3年业务演进，当前架构支持平滑扩展               │
│  3. 可观测     监控、告警、日志、链路追踪全链路覆盖                 │
│  4. 容错设计    任何组件故障不影响核心业务                          │
│  5. 成本意识   技术选型考虑人力、运维、基础设施成本                │
│  6. 团队适配   技术栈匹配团队能力，考虑学习曲线                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 3. 高可用设计

### 3.1 流量管控

```
┌─────────────────────────────────────────────────────────────────────┐
│                         流量管控策略                                 │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  限流 (Rate Limiting)                                               │
│  ┌──────────────────────────────────────────────────────────────┐  │
│  │  接口级别限流:                                                 │  │
│  │  - 查询接口: 1000 QPS/用户                                     │  │
│  │  - 搜索接口: 100 QPS/用户                                     │  │
│  │  - 写接口: 10 QPS/用户                                         │  │
│  │                                                                 │  │
│  │  全局限流:                                                     │  │
│  │  - 总 QPS: 10000                                               │  │
│  │  - 突发流量: 1.5x 容量                                         │  │
│  └──────────────────────────────────────────────────────────────┘  │
│                                                                      │
│  熔断 (Circuit Breaker)                                             │
│  ┌──────────────────────────────────────────────────────────────┐  │
│  │  - 慢调用比例 > 50% → 熔断打开                                │  │
│  │  - 错误比例 > 30% → 熔断打开                                  │  │
│  │  - 熔断时长: 30s                                              │  │
│  │  - 半开恢复: 3次成功则关闭                                    │  │
│  └──────────────────────────────────────────────────────────────┘  │
│                                                                      │
│  降级 (Degradation)                                                 │
│  ┌──────────────────────────────────────────────────────────────┐  │
│  │  - 搜索引擎故障 → 降级到数据库 LIKE 查询                     │  │
│  │  - 缓存失效 → 回源数据库                                     │  │
│  │  - 第三方服务 → 返回兜底数据                                   │  │
│  └──────────────────────────────────────────────────────────────┘  │
│                                                                      │
│  重试 (Retry)                                                       │
│  ┌──────────────────────────────────────────────────────────────┐  │
│  │  - 幂等写接口: 3次重试，指数退避                             │  │
│  │  - 读接口: 1次重试                                           │  │
│  │  - 重试间隔: 100ms → 200ms → 400ms                          │  │
│  └──────────────────────────────────────────────────────────────┘  │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 3.2 容灾策略

| 场景 | 策略 | RTO | RPO |
|------|------|-----|-----|
| 单机故障 | K8s 自动漂移 | 30s | 0 |
| 机房故障 | 多活/跨区部署 | 5min | 5min |
| 数据库故障 | 主从切换 | 30s | <1s |
| 缓存故障 | 降级 + 本地缓存 | 10s | - |
| 整体不可用 | 静态页面兜底 | 1min | - |

### 3.3 应急预案

```yaml
# 应急预案矩阵
emergency_plans:
  - scenario: "Redis 集群全挂"
    level: "P1"
    actions:
      - "自动降级: 关闭缓存，直接访问数据库"
      - "报警: 通知运维 + 架构师"
      - "恢复: 启动 Redis 备份实例"
    recovery_time: "10分钟"
    
  - scenario: "MySQL 主库故障"
    level: "P0"
    actions:
      - "自动切换: 从库提升为主库"
      - "报警: 通知 DBA + 架构师"
      - "恢复: 修复原主库，同步数据"
    recovery_time: "5分钟"
    
  - scenario: "搜索服务不可用"
    level: "P2"
    actions:
      - "自动降级: ES 查询降级为 MySQL LIKE 查询"
      - "报警: 通知搜索组"
      - "恢复: 重启 ES 节点"
    recovery_time: "5分钟"
```

## 4. 性能设计

### 4.1 缓存策略

```
┌─────────────────────────────────────────────────────────────────────┐
│                         多级缓存架构                                 │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   L1: 浏览器缓存                                                   │
│   └── 静态资源: JS/CSS/图片 (Cache-Control: max-age=31536000)     │
│   └── HTML: no-cache (必须验证)                                     │
│                                                                      │
│   L2: CDN 缓存                                                      │
│   └── 边缘节点: 全国分布                                             │
│   └── 缓存内容: 静态资源 + 热点接口 (如: 地区列表)                  │
│                                                                      │
│   L3: API 网关缓存                                                  │
│   └── 缓存内容: 热门招标列表 (TTL: 1分钟)                           │
│   └── 缓存 key: md5(请求参数 + 用户ID)                              │
│                                                                      │
│   L4: 应用本地缓存 (Caffeine)                                       │
│   └── 一级缓存: 热点数据 (TTL: 30秒, 大小: 10000)                  │
│   └── 缓存内容: 地区字典、热门搜索词、配置项                         │
│                                                                      │
│   L5: 分布式缓存 (Redis)                                            │
│   └── 二级缓存: 冷热数据 (TTL: 1小时)                              │
│   └── 缓存内容: 招标详情、用户会话、搜索结果                         │
│                                                                      │
│   L6: 数据库                                                        │
│   └── 持久化存储                                                     │
│   └── 索引优化: 地区 + 发布时间 + 招标类型                         │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 4.2 性能优化指标

| 接口 | 目标 P50 | 目标 P99 | 优化策略 |
|------|----------|----------|----------|
| 招标列表 | 50ms | 150ms | 缓存 + 分页 + 索引 |
| 招标详情 | 30ms | 100ms | Redis 缓存 |
| 搜索 | 100ms | 300ms | ES 分词 + 缓存 |
| 中标列表 | 50ms | 150ms | 缓存 + 分页 |
| 用户查询 | 20ms | 50ms | Redis 会话 |

## 10. 风险识别与应对

| 风险 | 影响 | 概率 | 应对策略 |
|------|------|------|----------|
| 数据量增长超预期 | 性能下降 | 中 | 提前做分库分表，预留扩展能力 |
| 爬虫被封禁 | 数据源丢失 | 高 | 多源 + 代理池 + 合规数据合作 |
| 搜索性能瓶颈 | 用户体验差 | 中 | ES 集群 + 读写分离 + 缓存 |
| 团队技术储备不足 | 交付延期 | 中 | 技术培训 + 专家咨询 + 渐进式引入 |
| 并发量突增 | 服务雪崩 | 中 | 限流 + 熔断 + 弹性扩容 |
| 安全攻击 | 数据泄露 | 低 | WAF + 渗透测试 + 安全审计 |
